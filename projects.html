Summary

Part of a team that developed a windows-based desktop application for users to log and analyze tasks related data.

Coordination

Split work on app by the app's pages (homepage, file creation page, etc.) to each members of the team so we could work in parallel.

Otherwise, progress on that section of the app could get behidn and be rushed at the last minute.

When someone is sick, make sure a backup developer has already been designated to take that person's spot.

Search Engine

Summary

Worked with a partner to develop backend of a search engine for our IN4 141 Search Engine project capable of handling 55,000+ documents or webpages.

Team Members:

  • Brendan Grant
  • Noah Wang

Document Relevance Scoring

Index

From a given document collection my partner and I designed our search engine to scrape tokens and stem them using Krovetz stemmer (due to speed and decent accuracy compared to Porter stemmer). Tokens are then stored in broken indexes (multiple text files) of generally 100-200MB each. Storing the tokens in one file would be too large. The data stored in each broken inverted index is a token followed by a list of docIDs (pointing a corresponding url) with that token's frequency, positions, and number of tokens that were titles, bolded, or headers in that given docID. After the broken indices are generated they are merged into a single merged inverted index and a separate reference index is kept which lists all discovered tokens and their memory locations in the merged index (to be used in hash table lookups when responding to queries).
